# PRD v1.0
# 청첩자리 — 청첩장 모임 장소 탐색 서비스

## 1. 서비스 개요

| 항목 | 내용 |
|---|---|
| 서비스명 | 청첩자리 |
| 서비스 유형 | 모바일 반응형 웹 |
| 한 줄 설명 | 예비부부가 관계와 모임 상황에 맞는 청첩장 모임 장소를 빠르게 찾고 저장할 수 있도록 돕는 장소 탐색 서비스 |
| 핵심 사용자 | 결혼식 1~3개월 전, 여러 관계의 청첩장 모임 장소를 반복해서 정해야 하는 예비부부 |
| MVP 지역 | 강남역, 신논현, 논현 인근 |
| 핵심 전환 | 장소 상세 확인 후 네이버지도/카카오맵 이동 |
| 개발 목적 | v1 MVP 구현 및 사용자 행동 데이터 수집 |
| 기술 스택 | Next.js, TypeScript, Tailwind CSS, Supabase, Mixpanel, Vercel |
| 디자인 톤 | 미니멀, 정갈함, 웜 뉴트럴, 정보 중심, 과하지 않은 웨딩 감성 |

---

## 2. 서비스 배경

- 청첩장 모임은 결혼식 전 1~3개월 사이에 여러 번 반복되는 일정이다.
- 예비부부는 친구, 직장 동료, 선배, 가족 등 관계별로 다른 장소를 선택해야 한다.
- 기존 지도 서비스는 위치, 리뷰, 사진, 메뉴 등 장소 중심 정보를 제공한다.
- 인스타그램, 블로그, 웨딩 커뮤니티에는 청첩장 모임 장소 추천 콘텐츠가 많다.
- 그러나 추천 리스트만으로는 다음 정보를 바로 판단하기 어렵다.
  - 어떤 관계의 모임에 적합한지
  - 왜 이 장소가 추천되는지
  - 예약이나 웨이팅 리스크가 있는지
  - 대화하기 좋은 분위기인지
  - 예산과 대접감이 적절한지
- 사용자는 추천 리스트를 본 뒤에도 네이버지도, 카카오맵, 블로그, 인스타그램을 다시 확인해야 한다.

## 3. 문제 정의

### 3.1 핵심 문제

예비부부는 청첩장 모임 장소를 고를 때 단순히 맛있는 식당이 아니라 관계와 모임 상황에 맞는 장소를 찾아야 한다.
그러나 기존 탐색 환경에서는 장소 판단 기준이 흩어져 있어 후보 검토와 비교에 시간이 오래 걸린다.

### 3.2 세부 문제

**문제 1. 관계별 장소 선정 기준이 다르다**

- 친한 친구 모임과 직장 선배 모임은 적합한 장소 기준이 다르다.
- 같은 식당이어도 관계에 따라 적합도가 달라진다.
- 사용자는 매번 "이 사람에게 이 장소가 괜찮을까?"를 다시 판단해야 한다.

**문제 2. 추천 리스트는 많지만 결정에 필요한 근거가 부족하다**

- 인스타그램, 블로그 등 SNS에는 청첩장 모임 장소 추천 콘텐츠가 존재한다.
- 그러나 추천 방식은 대부분 네이버지도/카카오맵의 지도 공유 리스트를 제공하며, 기존 맵 서비스 나열 방식인 장소명 중심으로 제공되는 경우가 많아 추천 이유를 바로 알기 어렵다.
- 사용자는 추천 리스트를 보고도 다시 리뷰, 블로그, 지도 정보를 검증해야 한다.

**문제 3. 결혼 준비 일정 안에서 반복 탐색을 감당하기 어렵다**

- 여러 그룹을 만나야 하고, 보통 결혼식 전 한두 달 사이에 약속이 몰린다.
- 하지만 이 시기엔 결혼식 준비로 바쁜 시기이다.
- 장소를 찾고, 비교하고, 저장하고, 예약 가능 여부를 확인하는 과정이 부담되며, 완벽한 장소보다는 실패 가능성이 낮은 후보를 원한다.

## 4. 제품 목표

### 4.1 사용자 목표

- 청첩장 모임에 적합한 장소 후보를 빠르게 찾는다.
- 관계와 상황에 맞는 추천 이유를 확인한다.
- 후보를 저장하고 다시 확인한다.
- 네이버지도 또는 카카오맵에서 최신 정보를 확인한다.

### 4.2 제품 목표

- 청첩장 모임 장소 선택 기준을 구조화한다.
- 장소 리스트가 아니라 판단 가능한 후보 카드를 제공한다.
- 사용자가 추가 검색 없이 후보를 1차 검토할 수 있게 한다.
- 지도 이동을 핵심 전환으로 측정한다.

### 4.3 v1 목표

- 강남권 장소 12~15곳 기반 MVP 구현
- 조건 입력 → 추천 결과 → 상세 → 저장 → 지도 이동 플로우 구현
- Mixpanel 이벤트 기반 퍼널 데이터 수집
- 추천 카드와 상세 화면의 정보 구조 검증

---

## 5. 핵심 가설

**가설 1. 관계를 먼저 기준으로 선택하면 장소 후보의 적합도를 빠르게 판단할 수 있다**

- 청첩장 모임은 관계에 따라 적합한 장소 기준이 달라진다.
- 사용자가 먼저 대상 관계를 선택하면, 일반 맛집 리스트보다 더 상황에 맞는 적합한 후보를 제공할 수 있다.

**가설 2. 추천 이유와 주의점을 함께 제공하면 재검색을 줄일 수 있다**

- 사용자는 장소명만 보고 결정하지 않는다.
- 추천 이유, 적합한 관계, 주의할 점을 함께 제공하면 사용자가 후보를 더 빠르게 판단할 수 있다.
- 사용자는 완벽한 맛집이 아닌, 실패하지 않을 청첩장 모임 식당을 원한다.

**가설 3. 공통 기준으로 후보를 비교하면 선택 부담이 줄어든다**

- 장소별 장단점을 청첩장 모임 기준으로 정리하면 비교가 쉬워진다.
- 대화 가능성, 예산 적합성, 예약 리스크, 술 여부, 위치 등을 공통 항목으로 제공한다.

## 6. MVP 범위

### 6.1 Must Have

v1에서 반드시 구현한다.

**1. 조건 입력**

사용자가 청첩장 모임 조건을 입력한다.

필수 입력:
- 대상 관계
- 인원
- 1인 예산
- 지역

선택 입력:
- 술 여부
- 원하는 분위기
- 피하고 싶은 조건

**2. 룰 기반 장소 추천**

- 사용자가 입력한 조건과 장소 DB 태그를 매칭한다.
- 매칭 점수 기준으로 장소 목록을 정렬한다.
- v1에서는 AI가 장소 순위를 결정하지 않는다.
- 추천 순위는 룰 기반 로직으로 계산한다.

**3. 추천 결과 목록**

추천 장소를 카드 형태로 보여준다.

카드에 포함할 정보:
- 장소명
- 카테고리
- 지역
- 가격대
- 적합도
- 추천 이유 한 줄
- 주요 태그
- 확인할 점
- 상세 보기 버튼
- 지도 보기 버튼

**4. 장소 상세**

사용자가 장소를 실제 후보로 검토할 수 있도록 상세 정보를 제공한다.

상세 화면에 포함할 정보:
- 장소명
- 카테고리
- 위치
- 가격대
- 이 모임에 추천하는 이유
- 청첩장 모임 체크포인트
- 주의할 점
- 네이버지도 링크
- 카카오맵 링크
- 북마크 버튼
- 추천 만족도 체크

**5. 북마크**

- 사용자가 마음에 드는 장소를 저장할 수 있다.
- v1에서는 로그인 없이 localStorage 기반으로 저장한다.
- 저장한 후보는 북마크함에서 다시 확인할 수 있다.

**6. 지도 이동**

- 장소 상세 또는 결과 카드에서 네이버지도/카카오맵으로 이동할 수 있다.
- 리뷰, 영업시간, 메뉴, 예약 가능 여부는 지도 서비스에서 확인하도록 한다.
- 지도 클릭을 MVP 핵심 전환으로 측정한다.

**7. 추천 만족도 체크**

- 장소 상세 화면에서 추천이 도움이 되었는지 응답할 수 있다.
- 응답 방식은 2가지로 제한한다.
  - 도움이 되었어요
  - 아쉬워요

### 6.2 Should Have

시간이 가능하면 구현한다.

- 조건 수정 기능
- 북마크함 진입
- 추천 만족도 이유 선택
- 후보 공유 링크
- 템플릿 기반 추천 이유 생성
- 모임별 후보 저장

### 6.3 Could Have

후순위로 둔다.

- 장소 비교 테이블
- 카카오톡 공유
- 배우자와 공동 후보 리스트
- 지도 임베드
- 강남 외 지역 확장
- 청첩장 모임 후기 수집
- AI 기반 추천 이유 문장화

### 6.4 Won't Have

v1에서는 제외한다.

- 예약 연동
- 결제
- 실시간 좌석 확인
- 실시간 웨이팅 확인
- 강남 외 지역 확장
- 사용자 계정 기반 프로필 관리
- 연락처 연동
- 웨딩 일정 관리
- 자동 장소 크롤링
- AI가 장소 순위까지 결정하는 추천 알고리즘

## 7. 사용자 플로우

### 7.1 기본 플로우

랜딩 → 조건 입력 → 추천 결과 목록 → 장소 상세 → 북마크 → 지도 이동

### 7.2 상세 플로우

1. 사용자가 랜딩 페이지에 진입한다.
2. CTA를 클릭해 조건 입력을 시작한다.
3. 청첩장 모임 조건을 입력한다.
4. 추천 결과 목록을 확인한다.
5. 장소 카드를 클릭해 상세 화면으로 이동한다.
6. 추천 이유와 체크포인트를 확인한다.
7. 마음에 드는 장소를 저장한다.
8. 네이버지도 또는 카카오맵으로 이동한다.

## 8. 화면 요구사항

### 8.1 화면 1. 랜딩

**목적**
- 서비스의 핵심 가치를 전달한다.
- 사용자가 조건 입력을 시작하도록 유도한다.

**주요 구성**
- 서비스명
- 핵심 카피
- 짧은 설명
- CTA 버튼
- 북마크함 진입 버튼

**카피 예시**
> 청첩장 모임, 어디서 해야 덜 고민될까요?

**서브카피 예시**
> 만나는 사람, 예산, 분위기, 피하고 싶은 조건을 기준으로 청첩장 모임에 맞는 장소를 찾아보세요.

**CTA**
> 모임 장소 찾아보기

### 8.2 화면 2. 조건 입력

**목적**
- 청첩장 모임의 상황을 입력한다.
- 추천 기준을 설정한다.

**입력 항목**

1. 대상 관계 (단일 선택)
   - 친한 친구
   - 친구 모임
   - 직장 동료
   - 선배/상사
   - 가족/친척
   - 기타

2. 인원 (단일 선택)
   - 2명
   - 3~4명
   - 5~6명
   - 7명 이상

3. 1인 예산 (단일 선택)
   - 2만원 이하
   - 2~3만원
   - 3~5만원
   - 5만원 이상

4. 술 여부 (단일 선택)
   - 술 없이 식사 중심
   - 술 함께 마시는 자리

5. 원하는 분위기 (복수 선택)
   - 조용히 대화하기 좋은
   - 너무 부담스럽지 않은
   - 적당히 대접하는 느낌
   - 캐주얼하지만 성의 있어 보이는
   - 오래 앉아 있기 좋은
   - 서비스가 안정적인

6. 피하고 싶은 조건 (복수 선택)
   - 웨이팅이 긴 곳
   - 예약이 너무 어려운 곳
   - 너무 시끄러운 곳
   - 너무 비싼 곳
   - 너무 가벼운 분위기
   - 너무 딱딱한 분위기
   - 역에서 먼 곳
   - 술집 느낌이 강한 곳

7. 지역 (단일 선택)
   - 강남역
   - 신논현
   - 논현
   - 강남권 전체

**버튼**
- 이전
- 다음
- 추천 보기

**유효성**
- 대상 관계, 인원, 예산, 지역은 필수값이다.
- 필수값 미선택 시 추천 보기 버튼은 비활성화한다.
- 선택 입력은 건너뛸 수 있다.

### 8.3 화면 3. 추천 결과 목록

**목적**
- 조건에 맞는 장소 후보를 빠르게 비교한다.
- 상세 확인, 저장, 지도 이동으로 이어지게 한다.

**주요 구성**
- 입력 조건 요약
- 조건 수정 버튼
- 추천 결과 수
- 장소 카드 리스트
- 북마크함 진입 버튼

**장소 카드 구성**
- 장소명
- 카테고리
- 지역
- 가격대
- 적합도 점수
- 추천 이유 한 줄
- 주요 태그 2~4개
- 확인할 점 1개
- 저장 버튼
- 상세 보기 버튼
- 지도 보기 버튼

**카드 카피 예시**
> 직장 동료 3~4명 모임에 무난해요.
> 대화하기 좋고, 가격대가 부담스럽지 않아요.
> 다만 주말 저녁은 예약 확인이 필요해요.

**빈 결과 문구**

조건에 맞는 장소가 없을 경우 다음 문구를 노출한다.
> 조건에 딱 맞는 장소를 찾지 못했어요.
> 피하고 싶은 조건을 줄이거나 지역을 넓혀보세요.

버튼:
- 조건 수정하기
- 강남권 전체로 보기

### 8.4 화면 4. 장소 상세

**목적**
- 사용자가 해당 장소를 실제 후보로 볼지 판단할 수 있도록 근거를 제공한다.

**정보 구조**

1. 기본 정보
   - 장소명
   - 카테고리
   - 위치
   - 가격대
   - 지역

2. 이 모임에 추천하는 이유
   - 사용자가 입력한 관계, 인원, 예산, 분위기 조건을 기반으로 추천 이유를 제공한다.
   - v1에서는 수동 작성 또는 템플릿 기반 문구를 사용한다.

   예시:

   > 이 장소는 역 접근성이 좋고, 3~4명이 대화하기 좋은 규모라 후보로 추천해요.

   - 대화 가능성

## 11. 추천 로직

### 11.1 기본 원칙

- v1에서는 룰 기반 매칭을 사용한다.
- AI는 추천 순위 결정에 사용하지 않는다.
- 사용자의 조건과 장소 DB 태그를 비교해 점수를 계산한다.
- 점수가 높은 순서대로 추천 결과를 보여준다.

### 11.2 점수 기준

총점 100점 기준

| 기준 | 점수 |
|---|---|
| 관계 적합도 | 25 |
| 인원 적합도 | 15 |
| 예산 적합도 | 15 |
| 분위기 적합도 | 15 |
| 술/무알콜 적합도 | 10 |
| 역 접근성 | 10 |
| 대화 가능성 | 10 |
| 피하고 싶은 조건 패널티 | -20 |

### 11.3 추천 결과 정렬

1. 1순위: 총점 높은 순
2. 2순위: 피하고 싶은 조건과 충돌이 적은 순
3. 3순위: 역 접근성 높은 순
4. 4순위: 대화 가능성 높은 순

### 11.4 추천 이유 생성

v1에서는 다음 중 하나로 구현한다.

**Option A. 수동 추천 이유**
- `places.curated_reason` 값을 그대로 보여준다.
- 가장 안정적인 방식이다.

**Option B. 템플릿 추천 이유**
- 사용자 조건과 장소 태그를 조합해 문장을 생성한다.
- 예시 템플릿:
  > {relationship}에게 청첩장을 전달하는 자리라면 {mood} 분위기의 장소가 적합해요.
  > 이 장소는 {strength_1}, {strength_2} 기준에서 잘 맞아 후보로 추천해요.
  > 다만 {caution_note}는 확인해보는 것이 좋아요.

**Option C. AI 추천 이유**
- v1 필수 아님.
- 룰 기반 추천 결과를 바탕으로 추천 이유만 자연어로 생성한다.
- 장소 순위 결정에는 사용하지 않는다.

## 20. 최종 방향

청첩자리는 맛집을 더 많이 보여주는 서비스가 아니다.

이미 사용자는 충분히 많은 맛집 리스트를 보고 있다. 문제는 그 리스트를 보고도 다시 검증해야 한다는 점이다.

따라서 v1의 목표는 장소 수를 늘리는 것이 아니라, 청첩장 모임에 맞는지 판단하기 쉽게 만드는 것이다.

핵심 방향:
- 관계 중심으로 탐색 시작
- 청첩장 모임 기준으로 장소 재태깅
- 추천 이유와 주의점 제공
- 후보 저장과 지도 이동 지원
- 지도 클릭 전환율로 MVP 효과 검증
