# PRD v1.2
# 청모픽 — 청첩장 모임 장소 탐색 서비스

> v1.2 개정 요약: 조건 입력을 9개 질문(4단계, 그중 1개는 선택 입력)으로 더 줄였다. 시간대 질문을 삭제하고, 예산 선택지를 MECE하게 재정의하고, 음식 카테고리 라벨을 지도·배달 앱 관례에 맞춰 다듬었다. 예약·웨이팅 질문을 하나의 직관적인 질문으로 합쳤고, 좌석·주차는 에어비앤비 스타일의 "기타 조건" 다중선택 칩으로 옮겼다. 우선순위 선택과 절대 기피조건 질문을 완전히 삭제했다. 입력 요약 화면도 없앴고, 그 대신 추천 결과를 준비하는 동안 조건을 하나씩 짚어주는 로딩 화면(마이크로 인터랙션)을 넣었다.

## 1. 서비스 개요

| 항목 | 내용 |
|---|---|
| 서비스명 | 청모픽 |
| 서비스 유형 | 모바일 반응형 웹 |
| 한 줄 설명 | 예비부부가 관계와 모임 상황에 맞는 청첩장 모임 장소를 빠르게 찾고 저장할 수 있도록 돕는 장소 탐색 서비스 |
| 핵심 사용자 | 결혼식 1~3개월 전, 여러 관계의 청첩장 모임 장소를 반복해서 정해야 하는 예비부부 |

## 2. 서비스 배경

- 청첩장 모임은 결혼식 전 1~3개월 사이에 여러 번 반복되는 일정이다.
- 예비부부는 친구, 직장 동료, 선배, 가족 등 관계별로 다른 장소를 선택해야 한다.
- 기존 지도 서비스는 위치, 리뷰, 사진, 메뉴 등 장소 중심 정보를 제공한다.
- 인스타그램, 블로그, 웨딩 커뮤니티에는 청첩장 모임 장소 추천 콘텐츠가 많다.
- 그러나 추천 리스트만으로는 다음 정보를 바로 판단하기 어렵다.
  - 어떤 관계의 모임에 적합한지
  - 왜 이 장소가 추천되는지
  - 예약이나 웨이팅 리스크가 있는지
  - 대화하기 좋은 분위기인지
  - 예산과 대접감이 적절한지
- 사용자는 추천 리스트를 본 뒤에도 네이버지도, 카카오맵, 블로그, 인스타그램을 다시 확인해야 한다.

## 3. 문제 정의

### 3.1 핵심 문제

예비부부는 청첩장 모임 장소를 고를 때 단순히 맛있는 식당이 아니라 관계와 모임 상황에 맞는 장소를 찾아야 한다.
그러나 기존 탐색 환경에서는 장소 판단 기준이 흩어져 있어 후보 검토와 비교에 시간이 오래 걸린다.

### 3.2 세부 문제

**문제 1. 관계별 장소 선정 기준이 다르다**

- 친한 친구 모임과 직장 선배 모임은 적합한 장소 기준이 다르다.
- 같은 식당이어도 관계에 따라 적합도가 달라진다.
- 사용자는 매번 "이 사람에게 이 장소가 괜찮을까?"를 다시 판단해야 한다.

**문제 2. 추천 리스트는 많지만 결정에 필요한 근거가 부족하다**

- 인스타그램, 블로그 등 SNS에는 청첩장 모임 장소 추천 콘텐츠가 존재한다.
- 그러나 추천 방식은 대부분 네이버지도/카카오맵의 지도 공유 리스트를 제공하며, 기존 맵 서비스 나열 방식인 장소명 중심으로 제공되는 경우가 많아 추천 이유를 바로 알기 어렵다.
- 사용자는 추천 리스트를 보고도 다시 리뷰, 블로그, 지도 정보를 검증해야 한다.

**문제 3. 결혼 준비 일정 안에서 반복 탐색을 감당하기 어렵다**

- 여러 그룹을 만나야 하고, 보통 결혼식 전 한두 달 사이에 약속이 몰린다.
- 하지만 이 시기엔 결혼식 준비로 바쁜 시기이다.
- 장소를 찾고, 비교하고, 저장하고, 예약 가능 여부를 확인하는 과정이 부담되며, 완벽한 장소보다는 실패 가능성이 낮은 후보를 원한다.

**문제 4. 조건마다 사용자가 느끼는 확신의 정도가 다르다**

- 사용자는 어떤 조건(예: 예산, 인원)에는 명확한 기준이 있지만, 다른 조건(예: 분위기, 예약)에는 뚜렷한 선호가 없다.
- 모든 문항에 억지로 답을 강요하면 입력 자체가 부담이 되고 이탈로 이어진다.
- 그렇다고 조건을 아예 생략하면 추천 정밀도가 떨어진다.

## 4. 제품 목표

### 4.1 사용자 목표

- 청첩장 모임에 적합한 장소 후보를 빠르게 찾는다.
- 관계와 상황에 맞는 추천 이유를 확인한다.
- 후보를 저장하고 다시 확인한다.
- 네이버지도 또는 카카오맵에서 최신 정보를 확인한다.

### 4.2 제품 목표

- 청첩장 모임 장소 선택 기준을 구조화한다.
- 장소 리스트가 아니라 판단 가능한 후보 카드를 제공한다.
- 사용자가 추가 검색 없이 후보를 1차 검토할 수 있게 한다.
- 지도 이동을 핵심 전환으로 측정한다.
- 꼭 필요한 질문만 남기고, 조건마다 확신의 정도가 다르다는 점을 인정해 입력 부담을 최소화한다.

### 4.3 v1 목표

- 강남권 장소 12~15곳 기반 MVP 구현
- 조건 입력(4단계 9개 질문, 그중 1개는 선택 입력) → 결과 준비 로딩 → 추천 결과 → 상세 → 저장 → 지도 이동 플로우 구현
- Mixpanel 이벤트 기반 퍼널 데이터 수집
- 추천 카드와 상세 화면의 정보 구조 검증
- 조건별 이탈률 · 추천 기여도 데이터를 축적해 v2 입력 항목 축소 기준을 마련

## 5. 핵심 가설

**가설 1. 관계를 먼저 기준으로 선택하면 장소 후보의 적합도를 빠르게 판단할 수 있다**

- 청첩장 모임은 관계에 따라 적합한 장소 기준이 달라진다.
- 사용자가 먼저 대상 관계를 선택하면, 일반 맛집 리스트보다 더 상황에 맞는 적합한 후보를 제공할 수 있다.

**가설 2. 추천 이유와 주의점을 함께 제공하면 재검색을 줄일 수 있다**

- 사용자는 장소명만 보고 결정하지 않는다.
- 추천 이유, 적합한 관계, 주의할 점을 함께 제공하면 사용자가 후보를 더 빠르게 판단할 수 있다.
- 사용자는 완벽한 맛집이 아닌, 실패하지 않을 청첩장 모임 식당을 원한다.

**가설 3. 공통 기준으로 후보를 비교하면 선택 부담이 줄어든다**

- 장소별 장단점을 청첩장 모임 기준으로 정리하면 비교가 쉬워진다.
- 대화 가능성, 예산 적합성, 예약 리스크, 위치 등을 공통 항목으로 제공한다.

**가설 4. 질문 수를 줄이고 '상관없어요'를 제공하면 입력 부담 없이 추천 정밀도를 지킬 수 있다**

- 꼭 필요한 질문만 남기고 나머지는 룩앤필이 아닌 실사용 데이터로 검증한다.
- 모든 조건에 명확한 선호를 강요하지 않으면 완주율이 올라간다.
- 룸·주차처럼 "있으면 좋은 정도"가 아니라 "꼭 필요한" 조건은 별도의 다중선택 칩으로 모아, 필요한 사람만 추가로 선택하게 한다.

## 6. MVP 범위

### 6.1 Must Have

v1에서 반드시 구현한다.

**1. 조건 입력 (4단계 9개 질문)**

사용자가 청첩장 모임 조건을 4단계에 걸쳐 입력한다. 전체 구조는 [8.2 조건 입력 개요](#82-화면-2-조건-입력-개요) 참조.

- Step 1. 모임 정보 — 관계, 인원, 지역 (3개, 필수)
- Step 2. 식사 조건 — 예산, 음식 종류 (2개, 필수)
- Step 3. 분위기·공간 — 대화 환경, 분위기·격식 (2개, 필수)
- Step 4. 운영 조건 — 예약·웨이팅(1개, 필수), 기타 조건(1개, 선택 · 다중선택 칩)

필수 질문에는 모두 '상관없어요'류 선택지를 제공해 명확한 선호를 강요하지 않는다. 기타 조건은 응답을 강제하지 않는 유일한 선택 입력이다.

**2. 룰 기반 장소 추천**

- 사용자가 입력한 조건과 장소 DB 태그를 매칭한다.
- 필수 조건(제외 조건)과 선호 조건(점수 반영 조건)을 구분해 처리한다.
- 매칭 점수 기준으로 장소 목록을 정렬한다.
- v1에서는 AI가 장소 순위를 결정하지 않는다.
- 추천 순위는 룰 기반 로직으로 계산한다.

**3. 결과 준비 로딩 화면**

- 조건 입력이 끝나면 바로 결과를 띄우지 않고, 입력한 조건을 하나씩 짚어주는 짧은 로딩 화면을 보여준다.
- "관계에 맞는 장소를 찾고 있어요" 같은 문구가 순차적으로 지나가는 마이크로 인터랙션으로 구성한다.
- 사용자가 자신의 입력이 반영되고 있다는 인상을 받도록 하고, 이후 결과 카드와의 연결감을 준다.

**4. 추천 결과 목록**

추천 장소를 카드 형태로 보여준다.

카드에 포함할 정보:
- 장소명
- 카테고리
- 지역
- 가격대
- 적합도 (매우 잘 맞아요 / 잘 맞아요 / 일부 조건을 확인해보세요)
- 추천 이유 한 줄
- 알고리즘이 판단한 핵심 강점 태그 2~3개
- 확인할 점 1개
- 상세 보기 버튼
- 지도 보기 버튼

**5. 장소 상세**

사용자가 장소를 실제 후보로 검토할 수 있도록 상세 정보를 제공한다.

상세 화면에 포함할 정보:
- 장소명
- 카테고리
- 위치
- 가격대
- 이 모임에 추천하는 이유
- 청첩장 모임 체크포인트
- 주의할 점
- 네이버지도 링크
- 카카오맵 링크
- 북마크 버튼
- 추천 만족도 체크

**6. 북마크**

- 사용자가 마음에 드는 장소를 저장할 수 있다.
- v1에서는 로그인 없이 localStorage 기반으로 저장한다.
- 저장한 후보는 북마크함에서 다시 확인할 수 있다.

**7. 지도 이동**

- 장소 상세 또는 결과 카드에서 네이버지도/카카오맵으로 이동할 수 있다.
- 리뷰, 영업시간, 메뉴, 예약 가능 여부는 지도 서비스에서 확인하도록 한다.
- 지도 클릭을 MVP 핵심 전환으로 측정한다.

**8. 추천 만족도 체크**

- 장소 상세 화면에서 추천이 도움이 되었는지 응답할 수 있다.
- 응답 방식은 2가지로 제한한다.
  - 도움이 되었어요
  - 아쉬워요

**9. 예외 상황 안내**

- 조건이 지나치게 좁아 후보가 1~2곳뿐일 때 결과 화면에서 안내하되 진행을 막지 않는다.
- 결과가 0개일 때 필수 조건은 유지한 채 완화 제안을 보여준다.
- 결과가 지나치게 많을 때 상위 후보를 우선 제공한다.

### 6.2 Should Have

시간이 가능하면 구현한다.

- 조건 수정 기능
- 북마크함 진입
- 추천 만족도 이유 선택
- 후보 공유 링크
- 템플릿 기반 추천 이유 생성
- 모임별 후보 저장

### 6.3 Could Have

후순위로 둔다.

- 장소 비교 테이블
- 카카오톡 공유
- 배우자와 공동 후보 리스트
- 지도 임베드
- 강남 외 지역 확장
- 청첩장 모임 후기 수집
- AI 기반 추천 이유 문장화

### 6.4 Won't Have

v1에서는 제외한다.

- 예약 연동
- 결제
- 실시간 좌석 확인
- 실시간 웨이팅 확인
- 강남 외 지역 확장
- 사용자 계정 기반 프로필 관리
- 연락처 연동
- 웨딩 일정 관리
- 자동 장소 크롤링
- AI가 장소 순위까지 결정하는 추천 알고리즘
- 정확한 날짜·시간·시간대 단위 입력 (모임 시점을 아예 묻지 않는다)

## 7. 사용자 플로우

### 7.1 기본 플로우

랜딩 → 조건 입력(4단계) → 결과 준비 로딩 → 추천 결과 목록 → 장소 상세 → 북마크 → 지도 이동

### 7.2 상세 플로우

1. 사용자가 랜딩 페이지에 진입한다.
2. CTA를 클릭해 조건 입력을 시작한다.
3. Step 1. 모임 정보(관계, 인원, 지역)를 입력한다.
4. Step 2. 식사 조건(예산, 음식 종류)을 입력한다.
5. Step 3. 분위기·공간(대화 환경, 분위기·격식)을 입력한다.
6. Step 4. 운영 조건(예약·웨이팅)을 입력하고, 필요하면 기타 조건 칩을 선택한다.
7. 마지막 질문에서 '추천 결과 보기'를 누르면 로딩 화면이 짧게 지나간다.
8. 추천 결과 목록을 확인한다.
9. 장소 카드를 클릭해 상세 화면으로 이동한다.
10. 추천 이유와 체크포인트를 확인한다.
11. 마음에 드는 장소를 저장한다.
12. 네이버지도 또는 카카오맵으로 이동한다.

## 8. 화면 요구사항

### 8.1 화면 1. 랜딩

**목적**
- 서비스의 핵심 가치를 전달한다.
- 사용자가 조건 입력을 시작하도록 유도한다.

**주요 구성**
- 서비스명
- 핵심 카피
- 짧은 설명
- CTA 버튼
- 북마크함 진입 버튼

**카피 예시**
> 청첩장 모임, 어디서 해야 덜 고민될까요?

**서브카피 예시**
> 만나는 사람, 예산, 분위기를 기준으로 청첩장 모임에 맞는 장소를 찾아보세요.

**CTA**
> 모임 장소 찾아보기

### 8.2 화면 2. 조건 입력 개요

**목적**
- 청첩장 모임의 상황을 4단계에 걸쳐 입력한다.
- 추천 기준(제외 조건 / 점수 반영 조건)을 설정한다.

**권장 플로우**

> 1. 모임 정보 → 2. 식사 조건 → 3. 분위기·공간 → 4. 운영 조건 → 결과 준비 로딩 → 추천 결과

| 단계 | 질문 수 | 목적 |
|---|---:|---|
| 1. 모임 정보 | 3개 | 누구와, 몇 명이, 어디서 만나는지 정의 |
| 2. 식사 조건 | 2개 | 예산과 메뉴 조건 정의 |
| 3. 분위기·공간 | 2개 | 청모 경험의 형태 정의 |
| 4. 운영 조건 | 1개 필수 + 1개 선택(다중선택 칩) | 예약·웨이팅과 특수 조건 정의 |

총 9개 질문 화면(그중 1개는 선택 입력)으로 구성한다.

**조건 역할 정의**

| 구분 | 역할 |
|---|---|
| 필수 조건 | 조건과 충돌하는 장소 제외 (인원·지역·예산 상한, 예약 필수, 기타 조건 칩) |
| 선호 조건 | 적합도 점수와 추천 순위에 반영 |

**공통 원칙**
- 필수 응답 질문에는 모두 '상관없어요'류 선택지를 제공해 명확한 선호를 강요하지 않는다.
- '조용한 분위기'와 '감성적인 분위기'처럼 성격이 다른 기준은 하나의 질문으로 묶지 않는다.
- 있으면 좋은 수준의 선호가 아니라 "꼭 필요한" 특수 조건(룸, 주차 등)은 그레이드가 있는 단일선택 질문 대신, 필요한 사람만 켜는 다중선택 칩으로 모은다.

**공통 버튼**
- 이전
- 다음
- (마지막 화면) 추천 결과 보기

### 8.3 화면 2-1. Step 1. 모임 정보

**2.1 누구에게 청첩장을 전달하나요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 친한 친구 / 친구 모임 / 직장 동료 / 학교·모임 선후배 / 직장 선배·상사 / 가족·친척 / 기타
- UX 문구: "누구와 만나는지에 따라 어울리는 장소가 달라져요."
- 추천 반영: 관계별로 장소 평가 가중치를 다르게 적용한다 ([11.5 관계별 가중치 보정](#115-관계별-가중치-보정) 참조).

**2.2 몇 명이 함께하나요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 2명 / 3~4명 / 5~6명 / 7~8명 / 9명 이상
- 데이터 형식: `people_count_min`, `people_count_max`
- 추천 반영:
  - 장소의 적합 인원 범위와 비교하고, 최대 수용 인원을 초과하면 결과에서 제외한다.
  - 5명 이상이면 예약 편의, 좌석 여유의 가중치가 증가한다.

**2.3 어느 지역에서 만나고 싶나요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 강남역 / 신논현역 / 논현역 / 강남권 전체
- 추천 반영: 특정 지역 선택 시 해당 지역을 우선하고, '강남권 전체' 선택 시 모든 지역을 접근성·적합도순으로 정렬한다.
- 예외 처리: 선택 지역에 후보가 부족하면 결과 화면에서 인접 지역 확장 여부를 안내한다. 사용자 동의 없이 지역을 자동 확장하지 않는다.

> **삭제한 질문**: '언제 만날 예정인가요?'(시간대)는 v1.2에서 삭제했다. 실시간 예약·웨이팅을 제공하지 않는 v1에서는 시간대별 정밀도가 입력 부담 대비 추천 정밀도 향상으로 이어지지 못한다고 판단했다. 장소의 소음·웨이팅 데이터는 시간대 구분 없이 단일 값으로 관리한다.

### 8.4 화면 2-2. Step 2. 식사 조건

**3.1 1인당 예산은 어느 정도인가요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 2만 원 이하 / 2만~3만 원 / 3만~5만 원 / 5만 원 이상 / 예산은 크게 중요하지 않아요

이전 안은 '5만~7만 원', '7만 원 이상', '가격은 크게 중요하지 않아요'가 함께 있어 상위 구간과 '상관없음'의 의미가 겹쳤다. v1.2에서는 상한 구간을 '5만 원 이상' 하나로 합치고, 구간을 반열림 구간(하한 포함, 상한 미포함)으로 정의해 경계값이 두 구간에 동시에 속하지 않도록 했다.

| 선택지 | 실제 매칭 구간 |
|---|---|
| 2만 원 이하 | 0원 ~ 20,000원 |
| 2만~3만 원 | 20,000원 ~ 30,000원 미만 |
| 3만~5만 원 | 30,000원 ~ 50,000원 미만 |
| 5만 원 이상 | 50,000원 ~ (상한 없음) |
| 예산은 크게 중요하지 않아요 | 모든 구간 중립 처리 |

- 추천 반영: 예산은 장소의 `price_min`, `price_max`와 비교한다 ([11.8 예산 매칭 규칙](#118-예산-매칭-규칙) 참조).
- 주의: 가격은 단일 메뉴 가격이 아니라 실제 모임에서 예상되는 1인 지출액을 기준으로 관리한다.

**3.2 어떤 음식을 선호하나요?**

- 입력 방식: 복수 선택, 최대 3개, 필수 응답('상관없음' 선택 가능)
- 선택지: 한식 / 양식 / 일식 / 중식 / 고기·구이 / 해산물 / 와인·주류와 함께 먹는 메뉴 / 브런치·카페 / 음식 종류는 상관없어요

네이버지도·카카오맵·배달의민족·쿠팡이츠가 공통으로 쓰는 업종 대분류(한식/양식/일식/중식/고기·구이/해산물 등)를 기준으로 라벨을 다듬었다. 기존 '고기'는 삼겹살인지 갈비인지 소고기인지 특정하지 않아 사용자가 무엇을 고르는지 헷갈릴 수 있어 '고기·구이'로 구체화했다.

- 추천 반영: 선택한 음식 종류와 장소의 음식 태그가 겹치는 비율만큼 가점하고, '상관없음'은 중립 처리한다. 관계별 메뉴 호불호 가중치도 함께 반영한다.

### 8.5 화면 2-3. Step 3. 분위기·공간

**4.1 어느 정도 조용한 장소를 원하나요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 대화가 잘 들리는 조용한 곳 / 적당히 활기 있지만 대화 가능한 곳 / 분위기와 활기가 더 중요해요 / 소음은 크게 중요하지 않아요
- 추천 반영: 장소의 `noise_level`과 `conversation_score`를 함께 평가한다.

**4.2 어떤 느낌의 모임을 원하나요?**

- 입력 방식: 단일 선택, 필수
- 선택지: 편안하고 캐주얼한 자리 / 너무 가볍지도 무겁지도 않은 자리 / 적당히 대접하는 느낌 / 격식 있고 차분한 자리 / 분위기는 크게 중요하지 않아요
- 추천 반영: 장소의 `hospitality_score`, `formality_level`, `mood_tags`와 비교한다.
- 관계와의 조합 예시: 친구+캐주얼 → 부담 없는 후보 우선 / 직장 상사+대접하는 느낌 → 서비스·분위기 점수 가중 / 가족+격식 있는 자리 → 대접감·메뉴 호불호 추가 반영

> **이동한 질문**: '어떤 좌석이 필요한가요?'는 v1.2에서 별도 질문으로 묻지 않는다. "꼭 룸이어야 한다"처럼 반드시 필요한 조건은 그레이드가 있는 단일선택이 아니라 [8.6의 기타 조건](#86-화면-2-4-step-4-운영-조건) 칩으로 옮겼다.

### 8.6 화면 2-4. Step 4. 운영 조건

**5.1 예약과 웨이팅은 어느 정도까지 괜찮나요?**

이전 안은 예약 방식(4지선다)과 웨이팅 허용(4지선다)을 한 화면에 나란히 둬 선택지가 총 8개로 과했다. v1.2에서는 두 축을 사용자가 실제로 갖는 태도 하나로 합쳐 4지선다 질문 하나로 줄였다.

- 입력 방식: 단일 선택, 필수
- 선택지:
  - 예약 필수예요, 웨이팅 없이 확실한 곳이 좋아요
  - 예약도 웨이팅도 어느 정도는 괜찮아요
  - 예약 없이 줄을 서더라도 괜찮아요
  - 예약·웨이팅은 크게 중요하지 않아요
- 추천 반영: 첫 번째 선택 시 예약이 어렵거나 불가능한 장소를 제외한다. 나머지는 예약 방식과 웨이팅 리스크를 함께 반영해 점수화한다 ([11.3](#113-점수-반영-조건) 참조).
- 표현 주의: 실시간 예약·웨이팅을 보장하지 않는다.
  > 현재 확인된 예약 방식과 일반적인 피크타임 기준이에요. 방문 전 최신 정보를 확인해주세요.

**5.2 기타 조건 (선택)**

에어비앤비 등 숙소 서비스의 편의시설 필터처럼, "있으면 좋은" 정도가 아니라 "꼭 필요한" 조건만 모아 다중선택 칩으로 제공한다. 아무것도 선택하지 않아도 다음으로 진행할 수 있는 v1의 유일한 선택 입력이다.

- 입력 방식: 다중 선택, 선택 입력(0개 이상)
- 선택지(칩):
  - 룸이 꼭 필요해요
  - 주차가 꼭 필요해요
  - 좌석 간격이 넉넉했으면 해요
  - 반분리된 좌석이면 좋겠어요
- 추천 반영: 선택한 칩은 모두 제외 조건으로 동작한다.

| 칩 | 처리 |
|---|---|
| 룸이 꼭 필요해요 | 룸이 없는 장소 제외 |
| 주차가 꼭 필요해요 | 주차가 불가능한 장소 제외 |
| 좌석 간격이 넉넉했으면 해요 | 좌석 여유 점수가 낮은 장소 제외 |
| 반분리된 좌석이면 좋겠어요 | 반분리·룸 형태가 아닌 장소 제외 |

- 주의: 룸이 있다는 정보만으로 추천하지 않는다. 룸 최소 인원, 최소 주문 금액, 이용 시간 제한, 룸 예약 방식은 상세 화면과 큐레이션 데이터에서 별도로 관리한다.

> **삭제한 질문**: '이번 모임에서 가장 중요한 조건은 무엇인가요?'(우선순위 선택)와 '절대 피하고 싶은 조건이 있나요?'(기피조건)는 v1.2에서 완전히 삭제했다. 우선순위에 따른 가중치 보정과 기피조건 감점/제외 로직도 함께 제거했다. 대신 알고리즘이 매 결과마다 자동으로 강점 태그를 뽑아 보여준다 ([12장](#12-결과-카드에-입력-조건을-연결하는-방식) 참조).

### 8.7 화면 3. 결과 준비 로딩

**목적**
- 입력이 바로 결과에 반영되고 있다는 인상을 준다.
- 별도의 요약 확인 화면 없이도, 사용자가 방금 고른 조건을 다시 한번 가볍게 인지하게 한다.

**구성**
- 스피너 또는 진행 인디케이터
- 1.5~2초 간격으로 바뀌는 짧은 문구, 사용자가 입력한 값을 문장에 끼워 넣는다.

**문구 예시 (순서대로 지나감)**
> {관계} 모임에 맞는 장소를 찾고 있어요…
> {예산} 예산에 맞는 후보를 좁히고 있어요…
> {음식 종류}이(가) 있는 곳을 확인하고 있어요…
> 대화하기 좋은 자리인지 살펴보고 있어요…
> 가장 잘 맞는 곳을 정리하고 있어요…

- 전체 노출 시간은 2~3초 내외로 제한한다. 실제 매칭 연산은 즉시 끝나므로, 이 화면은 연산 대기가 아니라 의도된 연출이다.
- 문구가 끝나면 자동으로 추천 결과 화면으로 전환한다(뒤로가기로 돌아오지 않도록 대체 이동한다).

### 8.8 화면 4. 추천 결과 목록

**목적**
- 조건에 맞는 장소 후보를 빠르게 비교한다.
- 상세 확인, 저장, 지도 이동으로 이어지게 한다.

**주요 구성**
- 입력 조건 요약
- 조건 수정 버튼
- 추천 결과 수
- (후보가 1~2곳으로 적을 때) 조건이 좁다는 안내와 완화 제안
- 장소 카드 리스트
- 북마크함 진입 버튼

**장소 카드 구성**
- 장소명
- 카테고리
- 지역
- 가격대
- 적합도 (매우 잘 맞아요 / 잘 맞아요 / 일부 조건을 확인해보세요)
- 추천 이유 한 줄
- 알고리즘이 판단한 핵심 강점 태그 2~3개 (예: `#대화하기 좋음` `#예산 적합` `#역 접근성 좋음`)
- 확인할 점 1개
- 저장 버튼
- 상세 보기 버튼
- 지도 보기 버튼

**카드 예시**

> **직장 동료 4명이 조용히 이야기하기 좋은 후보예요**
> 강남역에서 가깝고, 1인 3만~4만 원 선에서 식사할 수 있어요. 반분리 좌석이 있어 대화하기 좋습니다.
>
> `대화하기 좋음` `예산 적합` `예약 편의`
>
> **확인할 점**
> 주말 저녁에는 예약 가능한 좌석을 먼저 확인해보세요.

카드에서 보여주지 않는 것: 입력한 모든 조건, 모든 점수, 긴 AI 설명, 지도 서비스와 중복되는 리뷰·메뉴 정보.

**결과 0개 문구**

> 조건에 딱 맞는 장소를 찾지 못했어요.
> '룸 필수' 조건을 해제하면 4곳을 더 볼 수 있어요.

- 필수 조건은 최대한 유지한 채, 낮은 우선순위로 볼 수 있는 조건부터 완화를 제안한다(지역 → 예산 → 기타 조건 칩 순서로 하나씩 풀어보고 가장 효과가 큰 완화안을 보여준다).
- 시스템이 임의로 조건을 완화하지 않는다. 완화는 사용자가 버튼을 눌러야 적용된다.

버튼: 조건 수정하기 / 완화된 조건으로 보기(또는 강남권 전체로 보기)

**결과가 너무 많은 경우**

결과를 모두 보여주지 않고 상위 후보를 우선 제공한다.
> 입력한 조건에 잘 맞는 장소가 18곳 있어요. 우선 6곳을 추천해드릴게요.

### 8.9 화면 5. 장소 상세

**목적**
- 사용자가 해당 장소를 실제 후보로 볼지 판단할 수 있도록 근거를 제공한다.

**정보 구조**

1. 기본 정보 — 장소명, 카테고리, 위치, 가격대, 지역
2. 이 모임에 추천하는 이유 — 사용자가 입력한 관계, 인원, 예산, 분위기 조건을 기반으로 추천 이유를 제공한다. v1에서는 수동 작성 또는 템플릿 기반 문구를 사용한다.

   예시:
   > 직장 동료에게 청첩장을 전달하는 자리라면 너무 캐주얼한 술집보다 조용하게 식사할 수 있는 공간이 적합해요.
   > 이 장소는 역 접근성이 좋고, 3~4명이 대화하기 좋은 규모라 후보로 추천해요.

3. 청첩장 모임 체크포인트 — 대화 가능성, 예산 적합성, 인원 적합성, 역 접근성, 음식 적합성, 분위기·대접감, 좌석·공간, 예약 편의, 웨이팅 리스크, 서비스 안정감, 주차
4. 주의할 점

   예시:
   - 피크타임에는 소음이 있을 수 있어요.
   - 주말 저녁은 예약 확인이 필요해요.
   - 5인 이상이면 좌석 확인이 필요해요.

5. 지도에서 최신 정보 확인 — 네이버지도에서 보기 / 카카오맵에서 보기
   > 리뷰, 영업시간, 메뉴, 예약 가능 여부는 지도 서비스에서 최신 정보를 확인해주세요.

6. 액션 — 북마크, 추천 만족도 체크, 지도에서 보기

### 8.10 화면 6. 북마크함

**목적**
- 저장한 후보를 다시 확인한다.
- 저장한 후보에서 상세 화면 또는 지도 서비스로 이동한다.

**주요 구성**
- 저장한 장소 리스트
- 장소별 추천 이유 요약
- 가격대
- 지역
- 주요 체크포인트
- 상세 보기 버튼
- 지도 보기 버튼
- 북마크 해제 버튼

**빈 상태 문구**
> 아직 저장한 장소가 없어요.
> 마음에 드는 장소를 저장해두고 다시 비교해보세요.

## 11. 추천 로직

### 11.1 기본 원칙

- v1에서는 룰 기반 매칭을 사용한다.
- AI는 추천 순위 결정에 사용하지 않는다.
- 모든 조건을 동일한 방식으로 필터링하지 않는다. **필수 조건**은 결과 제외에, **선호 조건**은 점수 반영에 사용한다.
- 점수가 높은 순서대로 추천 결과를 보여준다.

### 11.2 결과 제외 조건

다음 중 하나라도 해당하면 추천 결과에서 제외한다.

- 인원 수용 불가
- 필수 지역 불일치
- 예산을 크게 초과
- 예약·웨이팅 질문에서 "예약 필수, 웨이팅 없이 확실한 곳"을 선택했는데 예약이 어렵거나 불가능한 장소
- 기타 조건 칩에서 "룸이 꼭 필요해요"를 선택했는데 룸이 없는 장소
- 기타 조건 칩에서 "주차가 꼭 필요해요"를 선택했는데 주차가 불가능한 장소
- 기타 조건 칩에서 "좌석 간격이 넉넉했으면 해요"를 선택했는데 좌석 여유 점수가 낮은 장소
- 기타 조건 칩에서 "반분리된 좌석이면 좋겠어요"를 선택했는데 반분리·룸 형태가 아닌 장소

### 11.3 점수 반영 조건

총점 97점 기준(v1.1에서 술 여부 3점을 제외했고, v1.2에서는 예약과 웨이팅을 하나의 질문·하나의 배점으로 합쳤다)

| 기준 | 기본 점수 |
|---|---:|
| 관계 적합도 | 15 |
| 인원 적합도 | 10 |
| 예산 적합도 | 15 |
| 위치·접근성 | 10 |
| 음식 적합도 | 10 |
| 대화 환경 | 10 |
| 분위기·대접감 | 10 |
| 좌석·공간 | 8 |
| 예약·웨이팅 | 7 |
| 주차 | 2 |
| 합계 | 97 |

- 점수는 내부 정렬에만 사용한다. 사용자에게 반드시 `87점`처럼 정밀한 숫자로 보여줄 필요는 없다.
- v1.1에 있던 "우선순위 1·2순위 가중치 보정"은 v1.2에서 삭제했다. 우선순위를 직접 고르게 하는 질문 자체가 없어졌기 때문이다. 대신 [11.5](#115-관계별-가중치-보정)·[11.6](#116-인원별-추가-가중치)의 관계·인원 기반 자동 보정만 남는다.

### 11.4 사용자 노출 방식

숫자 점수 대신 다음 3단계로 노출한다.

- 매우 잘 맞아요
- 잘 맞아요
- 일부 조건을 확인해보세요

숫자보다 왜 맞는지를 설명하는 것이 더 중요하다.

### 11.5 관계별 가중치 보정

| 관계 | 상대적으로 높게 반영할 기준 |
|---|---|
| 친한 친구 | 접근성, 가격, 캐주얼함, 좌석 여유 |
| 친구 모임 | 음식, 인원 수용, 접근성 |
| 직장 동료 | 접근성, 대화 환경, 가격, 예약 편의 |
| 학교·모임 선후배 | 음식, 분위기, 대화 환경 |
| 직장 선배·상사 | 대접감, 조용함, 예약 편의 |
| 가족·친척 | 음식, 좌석 여유, 주차, 대접감 |

### 11.6 인원별 추가 가중치

5명 이상일 경우 다음 항목의 가중치가 증가한다.
- 예약 편의
- 좌석·공간

### 11.7 지역 확장 로직

- 선택 지역에 후보가 부족하면 결과 화면에서 인접 지역 확장 여부를 안내 문구로 제시한다.
- 사용자 동의 없이 지역을 자동 확장하지 않는다.

### 11.8 예산 매칭 규칙

| 결과 | 처리 |
|---|---|
| 예산 범위 완전 일치 | 높은 점수 |
| 일부 겹침 | 중간 점수 |
| 예산보다 최대 10~15% 초과 | 낮은 점수와 주의사항 표시 |
| 크게 초과 | 결과 제외 |

구간은 하한 포함·상한 미포함의 반열림 구간으로 계산해([8.4](#84-화면-2-2-step-2-식사-조건) 표 참조), 경계값이 두 구간에 동시에 속하지 않도록 한다.

### 11.9 결과 정렬

1. 1순위: 총점 높은 순
2. 2순위: 역 접근성 높은 순
3. 3순위: 대화 가능성 높은 순

### 11.10 추천 이유 생성

v1에서는 다음 중 하나로 구현한다.

**Option A. 수동 추천 이유**
- `places.curated_reason` 값을 그대로 보여준다.
- 가장 안정적인 방식이다.

**Option B. 템플릿 추천 이유**
- 사용자 조건과 장소 태그를 조합해 문장을 생성한다.
- 예시 템플릿:
  > {relationship}에게 청첩장을 전달하는 자리라면 {mood} 분위기의 장소가 적합해요.
  > 이 장소는 {strength_1}, {strength_2} 기준에서 잘 맞아 후보로 추천해요.
  > 다만 {caution_note}는 확인해보는 것이 좋아요.

**Option C. AI 추천 이유**
- v1 필수 아님.
- 룰 기반 추천 결과를 바탕으로 추천 이유만 자연어로 생성한다.
- 장소 순위 결정에는 사용하지 않는다.

## 12. 결과 카드에 입력 조건을 연결하는 방식

사용자가 조건을 입력했다면, 결과에서 그 입력이 반영됐다는 보상이 보여야 한다. v1.1까지는 사용자가 직접 고른 "우선순위 조건" 충족 여부를 태그로 보여줬지만, 우선순위 질문이 삭제되면서 v1.2에서는 알고리즘이 매 결과마다 적합도 비율이 가장 높은 카테고리 2~3개를 자동으로 뽑아 강점 태그로 보여준다.

**카드에서 보여줄 것**
- 알고리즘이 판단한 핵심 강점 태그 2~3개 (적합도 비율 상위 카테고리)
- 추천 이유 한 줄
- 가장 중요한 주의점 1개
- 장소 기본 정보
- 상세 보기·저장하기

**카드에서 보여주지 않을 것**
- 입력한 모든 조건
- 모든 점수
- 긴 AI 설명
- 지도 서비스와 중복되는 리뷰·메뉴 정보

카드 예시는 [8.8 추천 결과 목록](#88-화면-4-추천-결과-목록) 참조.

## 13. 예외 상황

### 13.1 조건이 지나치게 좁은 경우

v1.1까지는 입력 요약 화면에서 결과 요청 전에 경고했지만, 입력 요약 화면이 삭제되면서 v1.2에서는 결과 화면 자체에서 안내한다. 후보가 1~2곳뿐이면 결과 화면 상단에 안내 문구를 함께 보여주되 진행을 막지 않는다.

> 조건에 맞는 장소가 많지 않아요. 아래 후보를 먼저 확인해보시고, 조건을 조정할 수도 있어요.

### 13.2 결과가 0개인 경우

- 필수 조건은 최대한 유지한다.
- 지역 → 예산 → 기타 조건 칩 순서로 하나씩 완화해보고, 가장 효과가 큰 완화안 하나를 제안한다.
- 시스템이 임의로 조건을 완화하지 않는다. 완화는 사용자가 버튼을 눌러야 적용된다.

예시:
> '룸 필수' 조건을 해제하면 4곳을 더 볼 수 있어요.

### 13.3 조건이 너무 넓은 경우

결과를 모두 보여주지 않고 상위 후보를 우선 제공한다.
> 입력한 조건에 잘 맞는 장소가 18곳 있어요. 우선 6곳을 추천해드릴게요.

## 16. UX/UI 원칙

### 16.1 정보 우선순위

- 장소명보다 추천 이유와 적합도를 빠르게 이해할 수 있어야 한다.
- 지도 서비스에서 확인 가능한 정보는 과하게 반복하지 않는다.
- 청첩장 모임 판단에 필요한 정보는 직접 구조화해 제공한다.

### 16.2 입력 부담 원칙

- 질문 수 자체를 최소화한다(4단계 9개 질문, 그중 1개는 선택 입력).
- 필수 질문에는 '상관없어요'류 선택지를 제공해 명확한 선호를 강요하지 않는다.
- "있으면 좋은" 선호와 "꼭 필요한" 특수 조건을 구분한다. 후자는 그레이드가 있는 단일선택 질문이 아니라 다중선택 칩(기타 조건)으로 모은다.
- 성격이 다른 기준(예: 소음과 감성적 분위기)은 하나의 질문으로 묶지 않는다.
- 정확한 날짜·시간·시간대처럼 추천 정밀도 대비 입력 부담이 큰 항목은 받지 않는다.

### 16.3 문체

- 사용자가 부담을 느끼지 않도록 부드러운 문장 사용
- 과장된 추천 표현 금지
- 확정적 표현보다 확인 유도 표현 사용
- 실시간 예약·웨이팅을 보장하지 않는다는 점을 명시한다.

예시: "예약이 필요해요" 보다 "주말 저녁은 예약 가능 여부를 먼저 확인해보세요"를 사용한다.

### 16.4 CTA

주요 CTA:
- 추천 결과 보기 (마지막 질문 화면의 버튼, 클릭 시 로딩 화면을 거쳐 결과로 이동)
- 상세 보기
- 저장하기
- 지도에서 보기

CTA 우선순위:
1. 추천 결과 보기
2. 상세 보기

### 20.2 v2 축소 판단 기준

v1에서는 위 9개 질문 구조를 구현하고, 이후 다음 기준으로 v2 축소 대상을 정한다.

## 21. 최종 방향

청모픽은 맛집을 더 많이 보여주는 서비스가 아니다.

이미 사용자는 충분히 많은 맛집 리스트를 보고 있다. 문제는 그 리스트를 보고도 다시 검증해야 한다는 점이다.

따라서 v1의 목표는 장소 수를 늘리는 것이 아니라, 청첩장 모임에 맞는지 판단하기 쉽게 만드는 것이다.

핵심 방향:
- 관계 중심으로 탐색 시작
- 청첩장 모임 기준으로 장소 재태깅
- 필수·선호 조건을 구분해 제외와 점수 반영을 다르게 처리
- 질문 수를 최소화하고, "꼭 필요한" 조건과 "있으면 좋은" 선호를 구분해서 받음
- 추천 이유와 주의점 제공
- 후보 저장과 지도 이동 지원
- 지도 클릭 전환율로 MVP 효과 검증
